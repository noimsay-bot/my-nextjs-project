# AGENTS.md

## 프로젝트 설명
이 프로젝트는 Next.js 15 App Router + Supabase + Vercel 기반의 사내 뉴스 영상팀 포털이다.
기존 구조를 최대한 유지하면서 필요한 기능만 안전하게 추가하는 것이 최우선이다.

## 작업 원칙
- 항상 최소 수정으로 해결한다.
- 사용자가 요청하지 않은 대규모 리팩터링은 하지 않는다.
- 기존 라우팅 구조, 권한 구조, 승인 흐름을 유지한다.
- 디자인보다 안정성, 권한, 빌드 성공을 우선한다.
- 추측으로 넓게 수정하지 말고 현재 코드 기준으로 보수적으로 수정한다.

## 작업 시작 전 반드시 할 일
1. 관련 파일을 먼저 찾는다.
2. 변경 범위를 짧게 요약한다.
3. 영향받는 페이지, 컴포넌트, 서버 로직을 적는다.
4. 기존 구조를 유지한 채 해결 가능한지 먼저 검토한다.

## 응답 형식
항상 아래 순서로 답한다.
1. 변경 대상 파일
2. 무엇을 왜 바꾸는지
3. 최소 수정안
4. 검증 방법
5. 필요한 SQL / env / 배포 반영사항

## 금지사항
- 요청하지 않은 전면 리팩터링 금지
- 인증/인가 로직 임의 단순화 금지
- RLS 우회 금지
- any 남발 금지
- 타입 에러를 숨기기 위한 무리한 캐스팅 금지
- 검증 없이 완료라고 단정 금지
- 서버/클라이언트 경계를 추측으로 변경 금지
- 기존 UI 전체를 갈아엎는 수정 금지

## 코드 규칙
- TypeScript 타입 안정성을 유지한다.
- 기존 네이밍과 폴더 구조를 최대한 따른다.
- 새 유틸은 lib 하위에, 새 UI는 components 하위에 둔다.
- page.tsx에는 페이지 조합 로직 위주로 두고 복잡한 UI는 components로 분리한다.
- 같은 작업 범위 안에서만 중복을 조심스럽게 줄인다.
- 주석은 꼭 필요한 경우에만 작성한다.
- 한글 문구는 기존 서비스 톤을 유지한다.

## Next.js 규칙
- App Router 구조를 유지한다.
- layout, page, route, server action의 역할을 섞지 않는다.
- 불필요하게 client 컴포넌트를 늘리지 않는다.
- hydration 비용이 커질 수 있는 전역 변경은 신중히 한다.
- 공개 페이지와 portal 라우트 그룹 구조를 존중한다.

## Supabase 규칙
- DB 스키마 변경이 필요하면 SQL 파일까지 함께 제안한다.
- RLS 영향 여부를 항상 함께 본다.
- auth, profiles, role, approved 관련 로직은 매우 신중히 다룬다.
- 프론트 수정만으로 해결 가능한 경우 DB를 먼저 건드리지 않는다.
- 운영 중인 데이터와의 호환성을 고려한다.

## 권한 및 보안 규칙
- admin, desk, team-lead, member 권한 로직은 절대 우회하지 않는다.
- approved 체크 흐름을 깨지 않는다.
- 민감 페이지는 UI 노출 여부뿐 아니라 데이터 접근 차단 여부까지 함께 확인한다.
- 서버에서 막아야 할 것은 서버에서 막는다.
- 클라이언트에서 숨기는 것만으로 보안이 해결됐다고 간주하지 않는다.

## UI/UX 규칙
- 모바일 화면 깨짐 여부를 항상 같이 점검한다.
- 홈 화면은 빠르고 단순하게 유지한다.
- 방송국 내부 포털 톤을 해치지 않는다.
- 관리자 화면은 기능 중심, 멤버 화면은 단순함 중심으로 유지한다.

## 성능 규칙
- 로그인, 미들웨어, 홈 화면, 스케줄은 성능 민감 영역이다.
- 불필요한 useEffect, 전역 상태, 재렌더 유발 코드를 피한다.
- prefetch, middleware, auth gate 비용을 의식한다.
- 가능하면 기존 최적화 패턴을 재사용한다.

## 검증 규칙
작업 후 반드시 아래를 점검한다.
- 타입 오류 가능성
- import 누락 여부
- 기존 페이지 영향 여부
- 권한별 접근 영향 여부
- 모바일 레이아웃 영향 여부
- npm run build 기준 검증 가능 여부

## 작업 우선순위
1. 빌드가 깨지지 않을 것
2. 권한/보안이 안전할 것
3. 기존 기능을 깨지 않을 것
4. 요청한 기능을 충족할 것
5. 구조를 최대한 유지할 것
6. 가독성을 개선할 것

## 리뷰 기준
- 권한 우회 가능성
- 타입 안정성
- 기존 기능 회귀 여부
- 모바일 UI 깨짐
- 불필요한 구조 변경
- Supabase/RLS 영향 누락

## 선호하는 작업 방식
- 먼저 관련 파일을 정확히 짚는다.
- 그 다음 최소 수정안을 제시한다.
- 필요할 때만 새 파일을 만든다.
- SQL이 필요하면 실행 파일명을 명확히 적는다.
- 최종 답변은 초보자도 적용 가능하게 경로와 파일명을 정확히 적는다.
